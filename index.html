ClickHouse 操作指南

本手册系统讲解 ClickHouse 业务元数据管理与备份恢复两大核心运维主题，覆盖原理、命令、参数、场景案例、故障排查与最佳实践，适用于 22.x 及以上版本，单机与集群场景通用。

PART 1 业务元数据操作指南 PART 2 备份操作指南 v1.0 适用 ClickHouse 22.x+

如何使用本指南

  • 左侧目录可快速跳转到任意章节，滚动时当前章节会自动高亮。
  • 顶部搜索框支持全文检索（按 / 快捷聚焦），输入关键词即时筛选匹配段落。
  • 所有命令块右上角均有「复制」按钮，便于直接粘贴使用。
  • 右上角打印按钮可一键导出 PDF 离线查阅。
  • 提示 蓝色框为补充信息；注意 黄色框为重要提醒；警告 红色框为高危操作前必读。

1业务元数据操作指南

1.1 概述与意义

什么是业务元数据

在 ClickHouse 中，元数据（Metadata）是描述数据的数据，包括：数据库与表的清单、表结构定义（列、类型、引擎、分区键、排序键、TTL）、分布表与副本拓扑、用户与权限、字典、视图与物化视图等。它们本身不存储业务数据，但决定了数据如何被组织、复制、查询与生命周期管理。

业务元数据则是在技术元数据之上叠加的业务语义，例如：表的中文注释、列的业务含义、数据归属团队、敏感等级、上下游血缘等。良好的业务元数据管理是数据治理、自助分析、合规审计的基础。

元数据管理的意义

  • 可发现性：分析人员能快速定位"我需要的数据在哪、长什么样"。
  • 可维护性：DDL 变更可追溯、可回滚，避免"幽灵表"堆积。
  • 可治理性：表/列注释、归属、敏感等级支撑数据分类分级与权限审计。
  • 可恢复性：元数据完整备份是数据恢复的前提——没有 schema，数据文件无法 attach。
  • 性能可控：分区/分片/TTL 设计直接影响查询性能与存储成本。
ℹ️ 关键认知

ClickHouse 的元数据是数据可读性的前提。一旦 /var/lib/clickhouse/metadata/ 损坏或丢失，即使数据 part 文件完好，也无法直接 attach 表。因此元数据备份与数据备份同等重要。

1.2 元数据存储机制

ClickHouse 的元数据分布在四个位置，理解它们的职责是排障与备份的基础：

存储位置内容类型变更方式
/var/lib/clickhouse/metadata/库/表的 DDL SQL 文件（.sql）持久化文件DDL 自动写入
system 数据库只读系统表（tables、columns、parts、clusters 等）内存/运行时查询自动反映
ZooKeeper / KeeperReplicated 表的副本分片状态、DDL 队列分布式协调引擎自动同步
配置文件 /etc/clickhouse-server/集群拓扑、macros、用户、磁盘XML/YAML手动编辑 + reload

SQL 元数据文件结构

text
# 每个数据库对应一个目录
/var/lib/clickhouse/metadata/
├── default/
│   └── hits.sql              # CREATE TABLE 的完整 DDL
├── analytics/
│   ├── events.sql            # 建库语句
│   └── logs.sql
└── system/                   # 系统库（不持久化）

# 文件内容示例 hits.sql
ATTACH TABLE hits (
  `id` UInt64,
  `event_time` DateTime,
  ...
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(event_time)
ORDER BY (id, event_time)
SETTINGS index_granularity = 8192;

注意文件使用 ATTACH TABLE 而非 CREATE TABLE——这是 ClickHouse 启动时重新挂载已存在数据 part 的机制。手工编辑这些文件极其危险，应通过 DDL 命令变更。

ZooKeeper 中的元数据路径

text
/clickhouse/tables/{shard}/{database}/{table}/
├── replicas/                # 副本清单与状态
│   ├── replica1/
│   │   ├── host
│   │   ├── pointer          # 当前最新 log pointer
│   │   └── is_active
│   └── replica2/
├── log/                     # DDL 操作日志队列
│   ├── log-0000000000
│   └── log-0000000001
└── columns/                 # 列定义快照

1.3 常用元数据查询

日常运维 90% 的元数据需求可通过 system 数据库完成。以下是高频查询模板：

数据库与表清单

sql
-- 列出所有数据库（含引擎、注释）
SELECT name, engine, comment, uuid
FROM system.databases
ORDER BY name;

-- 列出某库下所有表 + 引擎 + 行数 + 占用空间
SELECT
    name,
    engine,
    total_rows,
    formatReadableSize(total_bytes) AS size,
    metadata_modification_time AS last_ddl
FROM system.tables
WHERE database = 'analytics'
ORDER BY total_bytes DESC;

表结构查询

sql
-- 完整建表语句（最权威）
SHOW CREATE TABLE analytics.events;

-- 列信息（含类型、默认值、注释）
DESCRIBE analytics.events;
SELECT name, type, default_kind, default_expression, comment
FROM system.columns
WHERE database = 'analytics' AND table = 'events';

分区与数据 part

sql
-- 查看表的分区及每个分区大小
SELECT
    partition,
    formatReadableSize(sum(bytes_on_disk)) AS size,
    sum(rows) AS rows,
    count() AS part_count
FROM system.parts
WHERE database = 'analytics' AND table = 'events'
  AND active
GROUP BY partition
ORDER BY partition DESC;

-- 检查 mutation 进度（UPDATE/DELETE 异步任务）
SELECT database, table, mutation_id, command, is_done, parts_to_do
FROM system.mutations
WHERE NOT is_done;

集群拓扑与副本状态

sql
-- 集群分片副本拓扑
SELECT cluster, shard_num, replica_num, host_name, host_address, port
FROM system.clusters
WHERE cluster = 'my_cluster';

-- ReplicatedMergeTree 副本健康度
SELECT
    database, table,
    is_leader, is_readonly, is_session_expired,
    absolute_delay,   -- 落后主副本的 log 条数
    queue_size        -- 待执行的复制队列
FROM system.replicas
WHERE database = 'analytics'
ORDER BY absolute_delay DESC;
⚠️ 重点指标

absolute_delay 持续大于 0 表示副本落后；queue_size 持续增长表示复制队列堆积；is_readonly=1 通常意味着 ZooKeeper 会话失联，副本进入只读保护状态。

字典与视图

sql
-- 字典元数据
SELECT name, type, source, status, lifetime_min, lifetime_max
FROM system.dictionaries;

-- 物化视图依赖关系（找出物化视图读哪些表写哪些表）
SELECT database, name, as_select, to_table
FROM system.tables
WHERE engine = 'MaterializedView';

1.4 元数据变更操作

建库建表

sql
CREATE DATABASE analytics
ENGINE = Ordinary   -- 或 Atomic（默认，支持原子 DROP/RENAME）
COMMENT '业务分析库 - 归属数据团队';

CREATE TABLE analytics.events
(
    id          UInt64,
    user_id     UInt64 CODEC(Delta, ZSTD),
    event_time  DateTime CODEC(DoubleDelta, ZSTD),
    event_type  LowCardinality(String),
    props       Map(String, String),
    INDEX idx_user user_id TYPE bloom_filter GRANULARITY 1
)
ENGINE = MergeTree
PARTITION BY toYYYYMMDD(event_time)
ORDER BY (event_type, user_id, event_time)
TTL event_time + INTERVAL 90 DAY
SETTINGS index_granularity = 8192,
         ttl_only_drop_parts = 1
COMMENT '用户行为事件表 - 增量埋点';

列变更（ALTER）

sql
-- 新增列（默认追加到末尾，避免重写数据）
ALTER TABLE analytics.events
  ADD COLUMN channel String DEFAULT 'unknown' AFTER event_type;

-- 修改列类型（部分类型可原地转换，否则需重写）
ALTER TABLE analytics.events
  MODIFY COLUMN user_id UInt64 CODEC(Delta, ZSTD);

-- 重命名列
ALTER TABLE analytics.events RENAME COLUMN props TO properties;

-- 添加/修改列注释（业务元数据维护的关键）
ALTER TABLE analytics.events
  MODIFY COLUMN user_id COMMENT '用户唯一ID - 关联 users 表主键';

ALTER TABLE analytics.events MODIFY COMMENT '用户行为事件表 v2';

-- 删除列（注意会触发数据重写，大表慎用）
ALTER TABLE analytics.events DROP COLUMN deprecated_field;
⚠️ ALTER 注意事项

ClickHouse 的 ALTER 是异步元数据变更，命令返回不代表完成。需通过 system.mutations 跟踪进度。在 Replicated 表上还会通过 ZooKeeper 在副本间同步。删除列、修改列类型会触发数据重写，大表（TB 级）可能耗时数小时。

集群级 DDL（ON CLUSTER）

sql
-- 在集群所有节点上同步执行（Replicated 表只需在每分片一个副本执行）
CREATE TABLE analytics.events ON CLUSTER my_cluster (...)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/analytics/events',
    '{replica}'
)
...;

-- 查看分布式 DDL 队列状态
SELECT entry, host_name, status, exception
FROM system.distributed_ddl_queue
ORDER BY entry DESC LIMIT 20;

分区管理

sql
-- 卸载分区（数据保留在磁盘，元数据移除，表上看不到）
ALTER TABLE analytics.events DETACH PARTITION '20260101';

-- 重新挂载
ALTER TABLE analytics.events ATTACH PARTITION '20260101';

-- 跨表迁移分区（要求结构完全一致）
ALTER TABLE analytics.events
  MOVE PARTITION '20260101' TO analytics.events_archive;

-- 删除分区（不可逆，谨慎）
ALTER TABLE analytics.events DROP PARTITION '20250101';

1.5 分布式表与元数据

Distributed 表工作机制

Distributed 表本身不存储数据，它是一个"路由层"：保存目标集群名、分片权重、分片键、内部表名等元数据。查询时把请求下发到各分片的本地表，再聚合结果返回。

sql
-- 本地表（每个分片上创建）
CREATE TABLE analytics.events_local ON CLUSTER my_cluster (...)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/analytics/events_local',
    '{replica}'
)
PARTITION BY toYYYYMMDD(event_time)
ORDER BY (event_type, user_id, event_time);

-- 分布式表（在任一节点创建即可，所有节点都可见）
CREATE TABLE analytics.events_all ON CLUSTER my_cluster
AS analytics.events_local
ENGINE = Distributed(
    'my_cluster',        -- 集群名（来自 config）
    'analytics',         -- 目标库
    'events_local',      -- 本地表名
    rand()                -- 分片键（哈希分布）
);

macros 配置

上面的 {shard} 和 {replica} 是宏，需在每台服务器的配置文件中定义：

xml
<!-- /etc/clickhouse-server/config.d/macros.xml -->
<clickhouse>
    <macros>
        <shard>01</shard>
        <replica>node01</replica>
        <cluster>my_cluster</cluster>
    </macros>
</clickhouse>

不同节点配置不同值，DDL 模板就能在各节点展开为不同的 ZooKeeper 路径与副本名。

💡 命名约定

业界通行约定：本地表后缀 _local，分布式表后缀 _all（或 _distributed）。这能让查询者一眼分辨该表是物理表还是路由表。

1.6 元数据导出与迁移

导出 DDL

bash
# 导出整个库的所有表 DDL
clickhouse-client --query="
SELECT group_concat(ddl, ';\n') FROM (
  SELECT concat('DROP TABLE IF EXISTS ', database, '.', name, ';') AS ddl
  FROM system.tables WHERE database = 'analytics'
  UNION ALL
  SELECT show_create_table FROM system.tables WHERE database = 'analytics'
)" > analytics_schema.sql

# 或者使用原生工具（推荐）
clickhouse-client --dump-metadata --database=analytics > analytics_full.sql

# 导出单个表的 DDL
clickhouse-client --query="SHOW CREATE TABLE analytics.events FORMAT TabSeparatedRaw" \
  > events_ddl.sql

跨集群迁移流程

  1. 导出源端 DDL：使用上面的命令导出 schema。
  2. 修改引擎：如果目标集群拓扑不同，调整 ReplicatedMergeTree 的 ZooKeeper 路径与 macros。
  3. 在目标集群执行 DDL：用 ON CLUSTER 创建本地表与分布式表。
  4. 迁移数据：用 INSERT INTO ... SELECT 跨集群拉取，或用 clickhouse-backup 备份恢复，或用 remote() 表函数。
  5. 校验：对比两端的 system.parts 行数与校验和。
sql
-- 使用 remote() 表函数跨集群拉数据
INSERT INTO analytics.events_local
SELECT *
FROM remote('source-host:9000', analytics.events_local, 'user', 'password')
WHERE event_time >= '2026-07-01';

关键监控指标

指标来源告警阈值
表总数count() FROM system.tables趋势异常增长
每表 part 数system.parts> 300（合并跟不上）
Replicated 副本延迟system.replicas.absolute_delay> 1000 持续 5 分钟

元数据一致性修复

当副本元数据不一致（如某副本缺列、缺分区）时，按以下顺序处理：

sql
-- 1. 强制同步副本（从其他副本拉取缺失的元数据日志）
SYSTEM SYNC REPLICA analytics.events_local;

-- 2. 重启复制队列
SYSTEM RESTART REPLICA analytics.events_local;

-- 3. 严重不一致时，detach 后重新 attach（数据不丢，重新拉元数据）
DETACH TABLE analytics.events_local;
ATTACH TABLE analytics.events_local;

-- 4. 极端情况：删除副本注册，重新加入
SYSTEM DROP REPLICA 'node01' FROM TABLE analytics.events_local;
⛔ 高危操作

SYSTEM DROP REPLICA 会从 ZooKeeper 删除该副本的注册信息，仅用于副本已永久下线的场景。对仍在线的副本执行会导致该副本的数据被孤立。

1.8 最佳实践

命名规范

  • 库：snake_case，按业务域划分（如 analytics、user_center、billing）。
  • 表：{业务域}_{实体}_{粒度}，如 analytics_events_daily。
  • 本地表统一 _local 后缀，分布式表统一 _all 后缀。
  • 物化视图 mv_ 前缀，普通视图 view_ 前缀，字典 dict_ 前缀。
  • 避免使用 ClickHouse 保留字与中文。

注释规范

  • 库与表必须有 COMMENT，说明业务用途、归属团队、SLA。
  • 关键列必须有 COMMENT，说明业务含义、单位、取值范围。
  • 敏感字段（手机号、身份证）应注释其脱敏规则。
  • 定期审计：SELECT * FROM system.columns WHERE comment = '' 找出未注释列。

分区与分片设计

  • 分区键：按时间（toYYYYMM 或 toYYYYMMDD），便于 TTL 与按分区淘汰。单分区数据量建议 1GB~50GB。
  • 排序键：高基数过滤列在前，低基数列在后，避免把高基数列放最后。
  • 分片键：用 hash(user_id) 或 rand() 保证均匀；按用户分片利于点查。
  • 避免过度分区：单表分区数 > 1000 会拖慢 system.parts 查询与启动加载。

TTL 设计

sql
-- 热数据 30 天后移到 SSD，90 天后删除
ALTER TABLE analytics.events
  MODIFY TTL
    event_time + INTERVAL 30 DAY TO VOLUME 'cold',
    event_time + INTERVAL 90 DAY DELETE;

权限管理

sql
-- 基于角色的访问控制（RBAC，21.x+ 推荐）
CREATE ROLE analytics_reader;
GRANT SELECT ON analytics.* TO analytics_reader;

CREATE ROLE analytics_writer;
GRANT INSERT, SELECT ON analytics.* TO analytics_writer;

CREATE USER analyst_alice IDENTIFIED WITH sha256_password BY '***';
GRANT analytics_reader TO analyst_alice;

-- 限制资源（行数、内存、查询时长）
CREATE SETTINGS PROFILE analyst_profile
  SETTINGS max_rows_to_read = 100000000,
           max_execution_time = 60;
GRANT analyst_profile TO analytics_reader;

1.9 常见问题排查

Q1: "Metadata on replica is not up to date"

原因：副本启动时发现 ZooKeeper 中记录的列结构与本地 .sql 文件不符。

处理：

sql
-- 1. 确认是否本地元数据落后于 ZooKeeper
SELECT * FROM system.replicas
WHERE table = 'events_local';

-- 2. 强制拉取最新元数据
SYSTEM SYNC REPLICA analytics.events_local;

-- 3. 仍失败则 detach 后 attach（保留数据 part）
DETACH TABLE analytics.events_local;
ATTACH TABLE analytics.events_local;

Q2: ZooKeeper session expired，副本变只读

原因：网络抖动或 ZooKeeper 过载，会话超时。ClickHouse 进入只读保护，禁止写入。

处理：检查 ZooKeeper 健康（ruok / mntr），重启会话：

bash
# 检查 ZK 健康
echo ruok | nc zk1 2181
echo mntr | nc zk1 2181 | grep -E 'zk_watch_count|zk_outstanding_requests'

# 重启 ClickHouse 副本会话
clickhouse-client --query="SYSTEM RESTART REPLICA analytics.events_local"

# 若多个表都只读，重启 server（最后手段）
systemctl restart clickhouse-server

Q3: 表突然"丢失"

可能原因：DETACH 后未 attach；.sql 文件被误删；Atomic 库的表被延迟 drop。

排查：

bash
# 1. 检查数据 part 是否还在磁盘
ls /var/lib/clickhouse/data/analytics/events/

# 2. 检查 .sql 元数据文件
ls /var/lib/clickhouse/metadata/analytics/

# 3. 检查是否在 detached 区
ls /var/lib/clickhouse/data/analytics/events/detached/

# 4. 如数据还在但 .sql 丢了，可用 SHOW CREATE 在其他副本取 DDL 重建
clickhouse-client --query="SHOW CREATE TABLE analytics.events"

Q4: DDL 队列阻塞

原因：ON CLUSTER 命令在 ZooKeeper 的 /clickhouse/task_queue/ddl 排队，某个节点卡住会阻塞后续所有 DDL。

处理：

sql
-- 查看队列堆积
SELECT entry, host_name, status, exception, age
FROM system.distributed_ddl_queue
ORDER BY entry DESC LIMIT 10;

-- 强制终止卡住的 DDL（注意：仅终止等待中的，不影响已执行的）
SYSTEM KILL DISTRIBUTED DDL QUEUE;

-- 调整队列超时（避免单 DDL 卡死全集群）
# 在 config.xml 设置 distributed_ddl_max_timeout = 180

Q5: part 数过多，合并跟不上

sql
-- 检查 part 数
SELECT database, table, count() AS parts, sum(rows) AS rows
FROM system.parts
WHERE active
GROUP BY database, table
HAVING parts > 100
ORDER BY parts DESC;

-- 触发手动合并（针对某分区）
OPTIMIZE TABLE analytics.events FINAL;

-- 调整合并线程
# config.xml: background_pool_size = 16  (默认 16)
# 避免过于频繁的小写入，建议攒批 >= 10000 行/批

2备份操作指南

2.1 概述与备份策略

为什么必须备份

ClickHouse 经常被误以为"有副本就不用备份"。这是危险的误解。副本只能防御单节点硬件故障，无法防御：

  • 人为误操作：DROP TABLE、错误的 DELETE、TRUNCATE 会被同步到所有副本。
  • 逻辑损坏：错误的 ETL 写入脏数据，问题会复制到所有副本。
  • 勒索软件/安全事件：攻击者删除数据时副本同步生效。
  • 区域性灾难：整个机房失联。
  • 升级失败：新版本不兼容导致无法回滚。

RTO / RPO 概念

指标含义典型目标
RPO (Recovery Point Objective)可容忍的数据丢失量（时间维度）核心库 1 小时，分析库 24 小时
RTO (Recovery Time Objective)可容忍的业务中断时长核心库 1 小时，分析库 4 小时

3-2-1 备份原则

  • 3 份数据副本（生产 + 2 份备份）
  • 2 种不同介质（如对象存储 + 磁带/本地磁盘）
  • 1 份异地存放（跨区域/跨云）
⛔ 红线

副本 ≠ 备份。没有真实备份的 ClickHouse 集群，一次 DROP TABLE 误操作即可造成不可逆数据丢失。本指南的所有操作都以"备份必须独立于生产集群"为前提。

2.3 BACKUP / RESTORE 完整流程

ClickHouse 22.8 GA 的原生备份恢复命令，是当前最推荐的方式。它支持表/库/全库级别的备份、增量备份、S3/GCS/Azure 远程存储。

启用备份配置

xml
<!-- /etc/clickhouse-server/config.d/backup.xml -->
<clickhouse>
    <allowed_path>/var/lib/clickhouse/backups</allowed_path>
    <backups>
        <disk>
            <type>local</type>
            <path>/var/lib/clickhouse/backups</path>
        </disk>
        <disk>
            <type>s3</type>
            <endpoint>https://s3.amazonaws.com/my-backup-bucket/clickhouse/</endpoint>
            <access_key_id>AKIA...</access_key_id>
            <secret_access_key>***</secret_access_key>
        </disk>
    </backups>
</clickhouse>

本地全量备份

sql
-- 备份单表
BACKUP TABLE analytics.events
TO Disk('backups', 'events_20260717.zip');

-- 备份整库
BACKUP DATABASE analytics
TO Disk('backups', 'analytics_20260717.zip');

-- 备份全库（除 system 外）
BACKUP ALL
TO Disk('backups', 'full_20260717.zip');

-- 查看备份任务进度（异步执行）
SELECT id, status, num_files_processed, total_files, start_time, end_time
FROM system.backups
ORDER BY start_time DESC LIMIT 10;

备份到 S3

sql
BACKUP TABLE analytics.events
TO S3('https://s3.amazonaws.com/my-bucket/clickhouse/events_20260717.zip',
       'AKIA...', '***');

-- 带压缩（zstd 显著减小体积）
BACKUP TABLE analytics.events
TO S3('https://s3.amazonaws.com/my-bucket/clickhouse/events_20260717.zip',
       'AKIA...', '***')
SETTINGS compression_method = 'zstd',
         compression_level = 6;

增量备份

基于上一次全量或增量的备份基准，只备份变化部分。这是 BACKUP 命令的核心优势之一。

sql
-- 1. 周日做全量
BACKUP TABLE analytics.events
TO Disk('backups', 'events_full_20260717.zip');

-- 2. 之后每天做增量（基准指向上一次备份）
BACKUP TABLE analytics.events
TO Disk('backups', 'events_inc_20260718.zip')
SETTINGS base_backup = Disk('backups', 'events_full_20260717.zip');

BACKUP TABLE analytics.events
TO Disk('backups', 'events_inc_20260719.zip')
SETTINGS base_backup = Disk('backups', 'events_inc_20260718.zip');

RESTORE 恢复流程

sql
-- 恢复到原表（表必须不存在或结构兼容）
RESTORE TABLE analytics.events
FROM Disk('backups', 'events_full_20260717.zip');

-- 恢复到新表（用于验证或数据分析）
RESTORE TABLE analytics.events AS analytics.events_restored
FROM Disk('backups', 'events_full_20260717.zip');

RESTORE TABLE analytics.events AS analytics.events_restored
FROM Disk('backups', 'events_full_20260717.zip')
SETTINGS allow_different_database = 1,
         allow_different_table_def = 1;

-- 增量恢复：需按顺序应用全量 + 各次增量
RESTORE TABLE analytics.events
FROM Disk('backups', 'events_full_20260717.zip');

RESTORE TABLE analytics.events
FROM Disk('backups', 'events_inc_20260718.zip');

校验备份完整性

sql

-- 校验文件 checksum

✅ 推荐策略

每周日全量 + 每天增量 + 备份到 S3 跨区域，配合每月一次恢复演练，可达到 RPO=24h、RTO=2h 的标准。

2.5 ALTER TABLE FREEZE 传统方式

22.8 之前版本的主要备份手段。原理是利用文件系统的硬链接——把数据 part 链接到 /var/lib/clickhouse/shadow/，零拷贝。

操作步骤

sql
-- 1. 冻结整个表的所有分区
ALTER TABLE analytics.events FREEZE;

-- 2. 冻结指定分区
ALTER TABLE analytics.events FREEZE PARTITION '20260717';

-- 3. 冻结到指定备份名（区分多次备份）
ALTER TABLE analytics.events FREEZE WITH NAME 'backup_20260717';

使用 rsync 同步到备份服务器

bash
# 把 shadow 目录同步到备份机（硬链接→真实文件）
rsync -avz --progress \
  /var/lib/clickhouse/shadow/ \
  backup-server:/backups/clickhouse/20260717/

# 同步完成后清理 shadow 目录（重要！否则下次 FREEZE 会失败）
ls /var/lib/clickhouse/shadow/
rm -rf /var/lib/clickhouse/shadow/*

# 或使用官方清理命令
clickhouse-client --query="ALTER TABLE analytics.events UNFREEZE WITH NAME 'backup_20260717'"
⚠️ FREEZE 限制

1. 只支持本地，需自行 rsync 到远程；2. 不支持增量，每次都是全量；3. 占用 inode，大表会创建大量硬链接；4. shadow 目录不能残留，否则下次 FREEZE 报错；5. 不支持 RESTORE，恢复需手工 cp 回 data 目录后 ATTACH。

恢复流程（手工）

bash
# 1. 停表（detach 元数据，数据保留）
clickhouse-client --query="DETACH TABLE analytics.events"

# 2. 把备份的 part 拷回 data 目录
cp -r /backups/20260717/analytics/events/* \
  /var/lib/clickhouse/data/analytics/events/
chown -R clickhouse:clickhouse /var/lib/clickhouse/data/analytics/events/

# 3. 重新 attach
clickhouse-client --query="ATTACH TABLE analytics.events"

# 4. 校验数据
clickhouse-client --query="SELECT count() FROM analytics.events"

2.7 集群备份策略

分布式表备份策略

分布式表是路由表，不存数据，所以备份对象是本地表。常见两种策略：

  • 每分片独立备份：每个分片的本地表各自备份到独立的备份目标。优点：并行度高、网络流量分散；缺点：恢复时需协调多个分片。
  • 中心化统一备份：通过分布式表查询拉取全量数据到中心节点备份。优点：单一备份目标；缺点：流量集中、耗时长，不推荐大集群。

一致性考虑

分布式表的写入是异步的（通过 distributed_directory_monitor 批量分发），各分片的快照时间点可能不一致。对于强一致性要求的场景：

sql
-- 1. 暂停分布式表分发（让所有数据落到本地表）
SYSTEM STOP DISTRIBUTED SENDS analytics.events_all;

-- 2. 等待 pending 队列清空
SELECT count() FROM system.distribution_queue
WHERE database = 'analytics' AND table = 'events_all';
-- 等到为 0

-- 3. 各分片同时做备份（cron 或脚本协调）
BACKUP TABLE analytics.events_local TO ...

-- 4. 恢复分发
SYSTEM START DISTRIBUTED SENDS analytics.events_all;

跨区域容灾

对于核心业务，建议在异地部署一个 standby 集群，通过 ClickHouse 的 Cross-Replication 或 Kafka 双写实时同步。配合周期性全量备份到对象存储，构成"实时同步 + 定期快照"的两层防线。

2.8 灾难恢复演练

没有演练过的备份等于没有备份。建议每月执行一次恢复演练，每年执行一次完整灾备切换演练。

单节点恢复演练

  1. 在测试环境部署同版本 ClickHouse。
  2. 从备份恢复：先恢复 schema（DDL），再恢复数据。
  3. 校验：行数、校验和、抽样查询对比。
  4. 记录 RTO（从触发到恢复完成的时间）。
  5. 填写演练报告，归档。
bash
# 恢复演练脚本示例
set -e

BACKUP_NAME=$1
START_TIME=$(date +%s)

# 1. 恢复 schema
clickhouse-backup restore --schema $BACKUP_NAME

# 2. 恢复数据
clickhouse-backup restore --data $BACKUP_NAME

# 3. 校验
clickhouse-client --query="SELECT database, table, sum(rows) AS rows
                    FROM system.parts WHERE active GROUP BY database, table" \
  > /tmp/restored_counts.txt

diff /tmp/expected_counts.txt /tmp/restored_counts.txt && \
  echo "[OK] 行数校验通过" || echo "[FAIL] 行数不一致"

END_TIME=$(date +%s)
echo "RTO: $((END_TIME - START_TIME)) seconds"

集群级灾难切换演练

  1. 模拟主集群故障：停主集群的写入（应用切换写入开关）。
  2. 切换读取到 standby 集群：DNS / 配置中心切换。
  3. 验证业务读：跑核心查询，对比结果。
  4. 反向同步：故障恢复后，把切换期间的写入回灌主集群。
  5. 切回主集群。

2.10 最佳实践

备份频率

  • 核心 OLTP 库：每小时增量 + 每日全量。
  • 分析 OLAP 库：每日增量 + 每周全量。
  • 冷归档库：变更时备份。
  • 元数据（DDL/权限/配置）：每次变更即备份，且每日全量。

保留策略

  • 本地：7 天（快速恢复用）
  • 对象存储：30 天
  • 归档（Glacier/冷存储）：1 年
  • 合规要求场景：按法规保留 3~7 年

加密与访问控制

  • S3 启用 SSE-KMS 服务端加密。
  • clickhouse-backup 配置文件中的密钥用 KMS / Vault 管理，不要明文。
  • 备份 bucket 单独账号，最小权限，仅备份程序可写。
  • 恢复操作需双人复核（4-eye principle）。

生产环境备份脚本模板

bash
#!/bin/bash
# /opt/scripts/ch_backup.sh - 生产备份主脚本
set -euo pipefail

BACKUP_NAME="auto_$(date +%Y%m%d_%H%M)"
LOG=/var/log/clickhouse-backup.log
ALERT_WEBHOOK="https://hooks.slack.com/services/..."

log() { echo "[$(date '+%F %T')] $1" | tee -a $LOG; }

alert() {
  log "[FAIL] $1"
  curl -s -X POST $ALERT_WEBHOOK -H 'Content-Type: application/json' \
    -d "{\"text\":\"🚨 ClickHouse备份失败: $1\"}"
  exit 1
}

log "=== 备份开始: $BACKUP_NAME ==="

# 1. 创建本地备份
clickhouse-backup create $BACKUP_NAME || alert "create 失败"

# 2. 上传到 S3
clickhouse-backup upload $BACKUP_NAME || alert "upload 失败"

# 3. 元数据备份
clickhouse-client --query="SHOW CREATE TABLES FORMAT TabSeparated" \
  > /var/lib/clickhouse/backups/schema_$(date +%Y%m%d).sql || alert "schema 导出失败"

# 4. 清理旧备份
clickhouse-backup clean || log "[WARN] clean 失败，不影响备份"

# 5. 校验
BACKUP_SIZE=$(clickhouse-backup list $BACKUP_NAME | awk '{print $2}' | tail -1)
log "备份完成: $BACKUP_NAME, 大小: $BACKUP_SIZE"
